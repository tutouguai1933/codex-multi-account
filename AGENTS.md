## 语言与风格
用简洁、非技术的语言描述完成的内容，让非技术背景的人也能看懂。任务完成后给一个简短的总结，说清楚做了什么、结果是什么。中间过程如果要说，也用同样的风格，不需要列出每个技术细节。测试结果只报告通过/失败数量，不逐条列出用例。用中文回复我。

## 自主度
明确的子任务可以自主完成，不需要每步确认。遇到真正需要决策的分叉点再停下来问我。不要在任务完成后主动提出优化建议或替代方案。如果有重要问题需要注意，最多一句话注明，不展开。

## 文件处理
所有文件读写使用 UTF-8 编码，修改文件时不要改变原有编码。在 PowerShell 中读取含中文的文件时，先执行 `chcp 65001` 并设置 UTF-8 输出，读取时用 `Get-Content -Encoding UTF8`。不要用 sed/awk 处理含中文的文件，改用 Python 或 Node.js。代码注释使用中文。

## Python 环境
在项目中运行 Python 时，优先使用项目本地的虚拟环境。Windows 用 `.venv/Scripts/python.exe`，Unix 用 `.venv/bin/python`，不要直接用 `python` 命令。如果 `.venv` 不存在，先用 `python -m venv .venv` 创建再运行。

## 项目开始前
每次开始一个新项目或新会话时：
1. 先读 CONTEXT.md 了解当前进度和关键决定（如有），这是最轻量的启动方式
2. 根据任务需要再决定是否读 README.md 和 ARCHITECTURE.md，不需要每次都全读
3. 检查项目根目录是否有 AGENTS.md，有则优先读取其中的项目规则
4. 查看 README.md 里是否已有搜索记录，有则跳过搜索；没有且是新功能或新项目，才访问 https://skills.sh/ 和 GitHub 搜索可参考方案，并把搜索结论更新到 README.md 的搜索记录章节
5. 简单任务读完 CONTEXT.md 即可直接开始；复杂任务或新功能才需要读完整文档

## 项目记录
每次阶段性完成后同步更新以下文件：

**CONTEXT.md**（每次必须更新，保持极简）：
- 当前正在做什么
- 上次停在哪个位置
- 近期的关键决定和原因

**README.md** 和 **ARCHITECTURE.md** 满足以下任一条件才更新，日常小修小补不需要更新：
- 新增或删除了一个功能模块
- 部署方式、运行命令或依赖发生了变化
- 新增了重要的设计决定

**README.md** 内容包括：
- 项目功能简介（非技术语言）
- 技术架构（用了什么技术、各部分怎么配合）
- 本地运行方法
- 部署方法和命令
- 测试方法和常用命令
- 搜索记录（skills.sh 和 GitHub 的搜索结论，避免重复搜索）
- 已完成功能列表 + 待办事项

**ARCHITECTURE.md** 内容包括：
- 每个文件/模块的职责（一句话说清楚）
- 模块之间的调用关系
- 关键的设计决定和原因

## 代码习惯
- 调试 bug：先做完整的原因分析，包括上下游关联和是否有同类问题，再提出修改方案，不要直接跳到最小修改
- 涉及数据库操作或文件删除，必须先确认
- 需求不明确时，先问清楚再动手，不要自行猜测后大量编写

## Token 使用原则
避免重复运行任务。除非我明确要求，不要进行多轮实验、多次基准测试或自动调参。

## 代码规范
- 单个文件尽量控制在 500 行以内；超过 500 行时评估是否可以拆分成职责更清晰的模块，不强制但要有意识
- 函数保持单一职责，一个函数只做一件事，过长时主动拆分
- 每个文件顶部写一段简短注释，说明这个文件是干什么的、和哪些模块有关联
- 每个函数写一行注释说明用途，复杂逻辑在关键步骤处加注释解释意图
- 所有配置项、路径、密钥等不得硬编码，统一放配置文件管理
- 错误处理要完整，异常必须有明确提示或日志，不能静默失败
- 代码注释使用中文

## 前端交互验证
  - 只要修改了页面、按钮、表单、前端脚本或接口联动，不能只做代码检查或接口检查，必须验证真
    实页面返回内容和关键交互路径。
  - 对“点击按钮后发生什么”这类改动，必须至少验证：
    1. 页面返回的 HTML 已包含新增节点
    2. 前端脚本可正常解析
    3. 关键接口能返回预期结果
    4. 页面上的关键状态会发生变化，而不是只看后端日志
  - 如果本地已有旧服务占用端口，不能假设当前访问到的是最新代码；必须先确认端口对应进程，必
    要时停止旧进程后再验证。
  - 未完成真实交互验证前，不得说“已经修好”或“你现在可以试了”。
  - 前端问题默认按“代码审查 + 真实页面验证 + 关键点击路径验证”三步完成，缺一不可。
