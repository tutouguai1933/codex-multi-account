# codex-multi-account

`codex-multi-account` 是一个本地网页工具，用来统一管理 `OpenClaw` 和 `Codex CLI` 的账号切换。它把两个工具放进同一个账号池里，支持导入当前登录态、状态检测、手动切换、自动分流、禁用和删除。

## 技术架构

- 后端使用 `Python + FastAPI`
- 前端使用 `React + Vite + TypeScript`
- `OpenClaw` 通过读写 `~/.openclaw` 下的认证文件和配置实现切换
- `Codex CLI` 通过读写 `~/.codex/auth.json` 和本项目快照目录实现切换
- 项目自己的状态和事件保存在 `data/`
- 后端启动后会自己按设置周期自动刷新，不需要每次手动点按钮
- 前端构建完成后，后端可以直接托管网页，不必一直开着 `pnpm dev`
- 登录动作现在有独立状态流，页面里可以看到是否未开始、运行中、已收编或失败
- 页面里现在可以直接看到授权链接、最近输出摘要，支持取消登录，也能直接提交回调地址或授权码
- 服务重启后，上一次未完成的登录状态不会直接丢失
- 如果 CLI 回显了带授权码的回调地址，页面和本地状态会自动隐藏敏感参数
- 同一个真实账号仍然只有一份共享健康状态，但探测时会同时参考两侧绑定，不会因为单侧旧快照就把整账号误判成坏掉
- 用量检测继续使用 `wham/usage`，但现在会结合窗口时长和 reset 信息识别 5h 与 week，减少因为窗口顺序变化带来的误判
- 账户页支持批量导入 `cockpit-tools` 导出的 Codex JSON，导入后会同时补齐 `Codex CLI` 和 `OpenClaw` 两侧快照
- 统一账号池现在可以导出为兼容 `cockpit-tools` 的 Codex JSON，方便在 Windows 和 Linux 间倒账号
- 总览页现在是更偏运维控制台的布局，首页第一屏直接显示当前分配、风险概览、全局动作和最近结果
- 运行面板现在改成左右分区：左边直接看 OpenClaw / Codex 当前账号，右边看模式、自动刷新和最近执行，信息更直观，不再挤成一排
- 账号卡片现在只保留高价值信息：邮箱、Workspace、状态、额度和当前分配
- 卡片操作现在改成图标按钮，鼠标悬停再显示文字，首页更紧凑也更顺手
- 首页保留“检测全部额度”和“立即调度”两个主按钮
- 首页动作执行后会同步更新侧边栏状态；删除账号前会先弹确认；时间显示统一固定为中文 24 小时格式
- 调度按钮在真实运行时会先显示“正在调度”，等后端完成真实切换后再进入下一步反馈
- 总览和账号接口现在会自动脱敏，只把工作区、套餐、登录方式这类展示信息返回到网页，不会把导入时保存的 token 发给前端
- 总览页当前账号现在按机器上的真实运行态显示，不再只看旧分配缓存
- 手动点“立即调度一次”时，会强制重新挑选更健康且分流后的账号；后台自动轮询仍按阈值保守切换
- “一键检测所有额度”现在会立刻返回“已开始”，实际探测在后台完成，页面随后自动刷新，不会再把首页卡住
- `Codex CLI` 写回 `auth.json` 时，`last_refresh` 现在会写成 RFC 3339 时间字符串，避免新版 CLI 因时间格式报错

## 本地运行

先激活独立 conda 环境：

```bash
source /home/djy/miniforge3/etc/profile.d/conda.sh
conda activate codex-multi-account
```

启动后端：

```bash
cd /home/djy/codex-multi-account/backend
PYTHONPATH=/home/djy/codex-multi-account/backend/src python3 -m uvicorn codex_multi_account.app:create_app --factory --host 127.0.0.1 --port 9001
```

项目现在统一固定使用 `9001`，不再使用 `8080 / 18080` 这类临时端口。

开发期前端依赖需要你手动安装后再运行：

```bash
cd /home/djy/codex-multi-account/web
pnpm install
pnpm dev
```

预期结果：
- 后端启动后会提供 `/api/overview`、`/api/accounts`、`/api/settings`、`/api/events`
- 前端启动后会在浏览器里显示控制台页面
- 打开总览页后，可以先看到所有账号卡片，再看到自动刷新状态、刷新间隔和最近一次执行结果

开发期前端会把 `/api` 代理到本机 `9001`，这样可以直接连到当前后端服务。

如果要用更接近长期运行的单服务模式：

```bash
cd /home/djy/codex-multi-account/web
pnpm build
cd /home/djy/codex-multi-account/backend
PYTHONPATH=/home/djy/codex-multi-account/backend/src python3 -m uvicorn codex_multi_account.app:create_app --factory --host 127.0.0.1 --port 9001
```

预期结果：
- 浏览器直接打开 `http://127.0.0.1:9001/`
- 页面和接口都由同一个后端服务提供

## 部署方式和命令

第一版只支持本机运行，不做远程部署。现在已经支持两种方式：

- 开发模式：后端 + `pnpm dev`
- 单服务模式：前端 `build` 后只启动后端

仓库里已提供 `systemd` 安装脚本和登录启动脚本，便于在 WSL 里常驻运行。

## 已完成功能

- 后端适配 `OpenClaw` 当前登录态读取、快照保存、切换回写
- 后端适配 `Codex CLI` 当前登录态读取、快照保存、切换回写
- 统一账号池
- 手动导入、检测、切换、禁用、删除接口
- 设置接口和事件接口
- 分流优先的调度器骨架
- 后台自动刷新常驻运行
- 后台单次调度失败后会继续保活
- 自动调度同时考虑 5h 和 week 阈值
- 当前分配未变化时，调度结果会明确显示为 keep
- 没有 OpenClaw 候选账号时会记录明确事件
- 登录状态接口和页面状态面板
- 登录命令成功结束后，会尝试自动收编当前登录态
- 登录状态会持久化到本地，服务重启后仍可恢复显示
- 登录命令默认用 PTY 启动，避免 OpenClaw 因缺少 TTY 直接失败
- 账户页可直接提交回调地址或授权码给正在运行的登录流程
- 如果 CLI 回显回调地址，页面和 `data/login_sessions.json` 不会保留敏感 `code`
- 总览页显示自动刷新状态、最近执行结果
- 总览页现在会明确区分“未分配 / 部分分配 / 共用中 / 已分流”，不再把未分配误写成共用
- 同邮箱但不同 `account_id` 的导入条目不会再被误合并成一个账号
- 账户页新增批量导入 / 导出区，可直接粘贴 JSON 导入，也可一键导出兼容 JSON
- 总览页改成账号卡片墙，卡片里直接放邮箱、标签、额度和按钮
- 总览页已经从说明式首页改成双层运维面板，不再出现介绍型长文案
- 卡片操作已改成图标按钮，真实验证过悬停说明和点击后的状态变化
- 前端四页和真实接口接线
